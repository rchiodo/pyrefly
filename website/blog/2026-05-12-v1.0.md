Today we are pleased to share that Pyrefly, our open source type checker and language server for Python, has reached stable version 1 status, meaning we are confident that Pyrefly is ready for production use.

<!--truncate-->

Pyrefly is a Python code analysis tool for helping you find bugs in your code, provide structure for your AI agents and give you faster navigation in your IDE. It was first released as an alpha in mid-2025 and followed up with a beta in November of that year. Since then, we have shipped over 60 minor releases: fixing hundreds of bugs, adding the features you’ve been asking for, and improving performance to be one of the fastest tools out there.

Before we continue with this post we have to acknowledge that this would not have been possible without our amazing open-source community! To everyone who filed GitHub issues, submitted pull requests, gave us feedback at conferences, or joined us on Discord: thank you. Your contributions shaped this release, we’re grateful for every one of them, and we hope you continue being a part of the journey for future releases too.

So what does V1 actually mean for you? In this post, we will walk through what has changed since beta, what Pyrefly can do for your codebase today, and where we are headed next.

## Pyrefly is Production Ready

A question we get asked a lot is “when will Pyrefly be ready to use in production?”. Even before we officially bumped the version number, Pyrefly was already the default type checker for developers working on Instagram at Meta. It has been adopted by other large production codebases too, such as PyTorch, NumPy, Pandas (stubs) and JAX, where stability and performance are non-negotiable. And at the time of writing, the Pyrefly IDE extension is also the most downloaded extension on the [Open VSX registry](https://open-vsx.org/).

As the [Semantic Versioning FAQ](https://semver.org/?fbclid=IwY2xjawRe-aNleHRuA2FlbQIxMQBicmlkETFLUmZRckx3Zzl3ajlLd2Nhc3J0YwZhcHBfaWQBMAABHtUjUIlkI7F5LvTx9z-G8a0FaHUYI3WP579AlNhT5N9Q52fcra425xSCahjN_aem_ffD9QDOiZ-o9gx7QI-p4qA#how-do-i-know-when-to-release-100) puts it: "If your software is being used in production, it should probably already be 1.0.0.". So we thought it was time to make it official.

## Faster, More Accurate, More Reliable

From the start, our approach with Pyrefly was to aim for broad coverage of the Python typing spec and language semantics, even if that meant early releases surfaced more errors than users might have expected. We believed it was better to start comprehensive and refine, than to start narrow and surprise users with new errors over time. So if you tried Pyrefly early on and found it noisy, it is worth another look. Thanks to your ongoing feedback, we have been able to fix hundreds of bugs and address performance issues as soon as they come up, making each release of Pyrefly more performant than the last.

On the performance side, we have focused on making Pyrefly faster and leaner on large codebases. Since our beta release, a full type check on PyTorch is 34% faster, and incremental editor updates can be up to 125x faster. In real terms this means after an edit, Pyrefly now updates diagnostics for PyTorch in 19 milliseconds, down from 2.4 seconds[^1]. We have also invested in reducing false positives, so your editor is less noisy and the errors you do see are ones worth acting on.

To make sure we keep improving, we built an [open benchmarking suite](https://python-type-checking.com/typecheck_benchmark/) that runs every day against 53 popular open source Python packages. It tracks execution time and peak memory usage across Pyrefly and other popular typecheckers, so we can catch regressions early and share our progress transparently with the community.

## Works with Real-World Python codebases

No two Python projects look the same. You might be vibe coding a prototype in Antigravity, running type checks in CI across a massive codebase, or building a production API with Pydantic and Django. On the journey to a 1.0 release we’ve been focused on making Pyrefly adapt to the wide range of setups required by modern Python developers

A common frustration with existing type checkers is that your editor and your CI can give you different results. Pyrefly avoids this by providing both a CLI tool and an IDE extension, so you get consistent results everywhere. Over the past year, we have also seen a growing number of editors adopt Pyrefly, including new AI-forward IDEs, either [through dedicated extensions](https://www.youtube.com/shorts/GxzJu0NrUP0) or by making Pyrefly their [default language server](https://positron.posit.co/blog/posts/2026-03-31-python-type-checkers/). And we're currently excited to be working with Microsoft to bring Pyrefly support to Pylance, via a new Type Server Protocol (TSP). This integration, currently in early preview in the VSCode insiders build, allows users to choose Pyrefly for typechecking, while keeping Pylance features like integration with GitHub Copilot.

Pyrefly also now has built-in support for Pydantic and Django, including model validation, field types, and autocomplete on model fields. We have been steadily improving this support since their [first release](https://pyrefly.org/blog/pyrefly-beta/#preliminary-support-for-pydantic-and-django), and will continue to do so based on your feedback.

## Adopt at Your Own Pace

Whether you’re an experienced user of type checkers or just getting started adding types to your codebase, Pyrefly can meet you where you are.

If you’re ready to get going with the latest version of Pyrefly, you can quickly install the command-line tool:

```shell
pip install pyrefly
```

Or grab the IDE extension for your editor of choice from the [installation page](https://pyrefly.org/en/docs/installation/). You can also try Pyrefly in the browser with our [online sandbox](https://pyrefly.org/en/sandbox/) before installing anything.

If you’re installing Pyrefly for the first time without an existing type checker configuration, it uses the ***basic preset*** by default. This shows only high-confidence errors that are very likely to indicate real bugs or crashes, like syntax errors and missing imports. Other diagnostics are silenced. You can adjust your type checking mode in your VSCode settings or run `pyrefly init` to create a configuration file (`pyproject.toml`, `pyrefly.toml`), see the [configuration docs](https://pyrefly.org/en/docs/configuration/#preset) for preset details and other options.

### Migrating from Another Type Checker

If you are already using Mypy or Pyright, when you install Pyrefly it reads your existing configuration and does its best to match your configuration, so you do not have to configure everything from scratch. This won’t permanently save your Pyrefly configuration, so we recommend running `pyrefly init` to automatically create a `pyrefly.toml` (or update `pyproject.toml`) and save the settings.

```shell
pyrefly init
```

### Add Pyrefly incrementally

You don't have to type-check your entire codebase on day one. Use `project_excludes` in your \`pyrefly.toml\` or \`pyproject.toml\` to exclude specific areas of your codebase and only run Pyrefly on new code, then gradually bring legacy modules in over time.

```
project_excludes=['src/my/legacy/code']
```

### Gradually Adopting Types

If you have a codebase with little or no type coverage, Pyrefly can meet you where you are.  Try starting with one or two files and gradually expand type coverage over time.

When you are ready to start adding annotations explicitly, `pyrefly infer` can help with the heavy lifting. Run it from the command line to generate draft type annotations. For example, it can turn this:

```py
def total(numbers):
    result = []
    for n in numbers:
        result.append(n * 2)
    return sum(result)
```

into this:

```py
def total(numbers: list[int]) -> int:
    result = []
    for n in numbers:
        result.append(n * 2)
    return sum(result)
```

Depending on how your code is written it can sometimes be tricky to infer types correctly, so we recommend reviewing the output before merging.

Once you have started adding types, `pyrefly report` helps you track your progress. It produces a summary of annotation completeness and type completeness across your codebase, so you can measure where you stand and set goals for improvement over time.

### Tensor shape checking

One experimental feature we are particularly excited about is tensor shape checking for PyTorch and other ML frameworks. Shape mismatches are one of the most frustrating sources of runtime errors in machine learning code, and they can be expensive to discover when they crash a training run hours in.

Pyrefly's experimental tensor shape support lets you add a new form of type annotation to your models that includes shape information, so the type checker can track how tensor dimensions flow through your code and catch mismatches statically. Here is a simple example of a typed PyTorch model:

```py
import torch.nn as nn
from torch import Tensor
from shape_extensions import Dim # new type annotation extension from pyrefly

class BaselineActor[S, A](nn.Module):
    def __init__(self, state_size: Dim[S], action_size: Dim[A]) -> None:
        super().__init__()
        self.fc1 = nn.Linear(state_size, 400)
        self.fc2 = nn.Linear(400, 400)
        self.out = nn.Linear(400, action_size)

    def forward[B](self, state: Tensor[B, S]) -> Tensor[B, A]:
        h1 = F.relu(self.fc1(state))  # Pyrefly infers: Tensor[B, 400]
        h2 = F.relu(self.fc2(h1))     # Pyrefly infers: Tensor[B, 400]
        return torch.tanh(self.out(h2))  # Pyrefly infers: Tensor[B, A]

```

With just a few annotations at the class and function boundaries, Pyrefly can infer the shapes of all intermediate tensors and surface them as inlay hints in your editor. If a shape mismatch is introduced, say by connecting two layers with incompatible dimensions, Pyrefly catches it before you ever run the code.

This feature is still experimental, and we want to actively engage with the Python typing community on how to improve it and take this work further. You can check out the documentation [here](https://pyrefly.org/en/docs/tensor-shapes/) for more information.

Growing codebases have always benefitted from the safety net of static types, and the volume of AI-assisted code being written today makes this more relevant than ever.
